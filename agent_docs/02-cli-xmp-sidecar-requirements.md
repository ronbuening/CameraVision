# Phase 2 Requirements - CLI XMP Sidecar Writer

Version: 0.2
Date: 2026-06-10
Supersedes: 0.1
Binary: `aisidecar` (subcommand: `write-xmp`)
Core library: `AISidecarCore`
Minimum deployment target: macOS 15
Default vision model: `gemma4:26b-a4b-it-qat`
Primary output artifact: XMP sidecar file

This document inherits the Project-Wide Conventions of the Phase 1 requirements (Section 1 there): binary/subcommand structure, flag glossary, configuration resolution, error taxonomy, schema evolution, provenance principles, and concurrency model. They are not restated here.

## 0. Changes from v0.1

1. The tool is the `write-xmp` subcommand of `aisidecar`, not a separate binary (PW-001).
2. Confidence filtering operates on ordinal bands (`--min-confidence <low|medium|high>`), matching the Phase 1 schema; the float form is removed.
3. The metadata write engine is specified as a `MetadataWriteEngine` protocol with ExifTool in `-stay_open` batch mode as the first implementation; ExifTool presence and version are verified at startup and recorded in provenance (FR2-029a-c).
4. RAW+JPEG shared-sidecar merge semantics are defined: union of candidates with per-source provenance, single merge-write per XMP path (FR2-002a-c).
5. The undefined "confirmation file" mechanism is removed; `--allow-specific-tags` is the only Phase 2 override, and Phase 3 session files are the structured mechanism.
6. Keyword text rules are defined: NFC normalization, trimming, whitespace collapsing, and rejection of terms containing the hierarchy separator (FR2-006a-c).
7. Post-write validation now requires a round-trip preservation diff, with defined failure behavior (FR2-028, FR2-028a).
8. Export reports must include post-export application instructions for Lightroom Classic and Capture One (FR2-034a), promoted from Phase 4 because Phase 2 is where the CLI workflow will actually live for months.

## 1. Purpose

Phase 2 shall expand the toolchain so it can write photographic metadata into XMP sidecar files. It shall analyze the image as Phase 1 does, extract candidate tags from the model output, and write a standards-oriented XMP sidecar that can later be read by Lightroom Classic, Capture One, and other metadata-aware applications.

This phase remains command-line only. It does not yet provide cross-image normalization, a GUI review queue, or a controlled-vocabulary editor.

## 2. Builds Upon Phase 1

Phase 2 shall reuse the following Phase 1 modules from `AISidecarCore` without rewrites:

- file/folder scanner and source identity;
- derivative rendering, orientation/color rules, and cache;
- the subject-isolation pipeline;
- model input profiles;
- the Ollama model adapter with mock and recorded-fixture runners;
- `--mode whole|subject|both` analysis;
- the raw JSON sidecar schema with banded candidates;
- provenance handling;
- JSONL progress logging, batch summaries, and error taxonomy.

Phase 2 may either run analysis and XMP export in one pass or consume existing `.ai.json` sidecars generated by Phase 1.

## 3. Scope

Phase 2 adds XMP sidecar generation and update behavior.

The subcommand shall support two workflow styles:

```text
Analyze-and-write:
  aisidecar write-xmp <file-or-folder>           image -> model run -> extraction -> XMP

Write-from-json:
  aisidecar write-xmp --from-json <path>         existing .ai.json -> extraction -> XMP
```

## 4. Command-Line Interface Requirements

Required command shapes:

```bash
aisidecar write-xmp <file-or-folder> --mode both

aisidecar write-xmp --from-json <json-file-or-folder>
```

Accepted flags: the project-wide glossary (PW-004) plus:

```text
--from-json <path>
--write-flat-keywords / --no-write-flat-keywords
--write-hierarchical-keywords / --no-write-hierarchical-keywords
--backup-sidecars / --no-backup-sidecars
--xmp-conflict-policy <fail|merge|backup-and-merge>
--min-confidence <low|medium|high>
--allow-specific-tags
--pair-scope <union|raw-only|jpeg-only>
--write-ai-json / --no-write-ai-json
```

Default behavior shall be conservative:

```text
--write-flat-keywords           enabled
--write-hierarchical-keywords   enabled
--dry-run                       disabled
--backup-sidecars               enabled
--xmp-conflict-policy           backup-and-merge
--min-confidence                medium
--allow-specific-tags           disabled (specific tags excluded)
--pair-scope                    union
--write-ai-json                 enabled
```

## 5. XMP Sidecar Naming and RAW+JPEG Pair Requirements

FR2-001 - For proprietary RAW files, the XMP sidecar path shall use the source base filename and `.xmp` extension:

```text
_DSC1234.NEF -> _DSC1234.xmp
_DSC1234.RAF -> _DSC1234.xmp
_DSC1234.ARW -> _DSC1234.xmp
```

FR2-002 - Same-base-name files with different extensions share one XMP sidecar in Lightroom Classic and Capture One. The program shall detect such pairs (or larger groups) during scanning, before any export begins.

FR2-002a - Merge semantics for shared sidecars: by default (`--pair-scope union`), the candidate sets from all members of the group (e.g., `_DSC1234.NEF.ai.json` and `_DSC1234.JPG.ai.json`) shall be unioned, de-duplicated case-insensitively after text normalization (FR2-006a), with per-candidate provenance recording which source image produced each tag. `--pair-scope raw-only` and `--pair-scope jpeg-only` restrict extraction to one member.

FR2-002b - Exactly one merge-write shall be performed per XMP path per batch. Interleaved or sequential per-member writes to the same sidecar are forbidden; the group is resolved to a single write plan first.

FR2-002c - The export report shall list every detected group, the scope applied, and the per-source contribution counts.

FR2-003 - Detected groups shall produce a warning in the report; the `--xmp-conflict-policy` continues to govern conflicts with pre-existing sidecar content, which is a separate question from group merging.

FR2-004 - Phase 2 shall not modify proprietary RAW files. Source file hashes shall be verifiably unchanged after any run (AC2-007).

FR2-005 - Embedded metadata writing to JPEG/TIFF/DNG is out of scope for the MVP; behavior is sidecar-only for consistency. This decision shall be revisited no earlier than Phase 4.

## 6. Metadata Mapping Requirements

FR2-006 - Accepted flat keywords shall be written to `XMP-dc:Subject`.

FR2-006a - Keyword text normalization, applied before de-duplication and writing: Unicode NFC normalization; leading/trailing whitespace trimmed; internal whitespace runs collapsed to single spaces. Model casing is preserved in Phase 2 (canonical casing is a Phase 3 vocabulary concern).

FR2-006b - Terms containing the Lightroom hierarchy separator `|` shall be rejected from export entirely — flat and hierarchical — with a warning in the report. Silently mangling a term is worse than dropping a tag the user can add by hand; model output will eventually contain a stray pipe and the behavior must be defined.

FR2-006c - Empty terms after normalization shall be dropped silently.

FR2-007 - Accepted hierarchical keywords shall be written to `XMP-lr:HierarchicalSubject` when hierarchical export is enabled, using `|` as the level separator.

FR2-008 - Caption/description writing shall be disabled by default. If enabled, it shall write only user-approved generated text to `XMP-dc:Description`.

FR2-009 - Title writing shall be disabled by default. If enabled, it shall write only user-approved generated text to `XMP-dc:Title`.

FR2-010 - Rating and color-label writing shall be disabled by default.

FR2-011 - The program shall not write AI-inferred camera, lens, date, exposure, GPS, named-person, or exact-location facts.

FR2-012 - The program shall not write Adobe Camera Raw, Lightroom develop, or Capture One adjustment settings.

## 7. Candidate Extraction Requirements

FR2-013 - The program shall extract candidate keywords from the Phase 1 model JSON using the Phase 1 response schema (FR1-044/045), respecting the schema-evolution rule (PW-012).

FR2-014 - If both whole-image and subject-isolated runs exist, their contributions shall remain distinct in the intermediate extraction record.

FR2-015 - Whole-image candidates shall be preferred for scene, setting, habitat, lighting, background, and broad photographic context. The subject-isolated schema cannot produce these fields by construction (FR1-045).

FR2-016 - Subject-isolated candidates shall be preferred for subject morphology, object/species detail, and fine subject classification.

FR2-017 - Phase 2 shall not perform cross-image normalization. De-duplication is within a single image (or within a same-base-name group under FR2-002a) only.

FR2-018 - Candidates below `--min-confidence` shall not be exported. The band ordering is `low < medium < high`; the default threshold is `medium`. There is no numeric confidence anywhere in the system (FR1-044) and none shall be introduced here.

FR2-019 - Specific tags — species names, named places, named events, named people, rare species, and exact-location implications — shall be excluded from export unless `--allow-specific-tags` is supplied. The v0.1 "confirmation file" is removed: a second, undefined mechanism is worse than one defined flag, and Phase 3 normalization session files are the structured per-tag mechanism for this purpose.

FR2-019a - Specific-tag detection in Phase 2 is heuristic (capitalized multi-word terms, binomial patterns, terms flagged by the model's own evidence text) and shall err toward exclusion. Phase 3's vocabulary `requires_review` field replaces the heuristic with policy.

## 8. XMP Merge Requirements

FR2-020 - Existing XMP sidecars shall be read before writing.

FR2-021 - Existing flat keywords shall be preserved by default.

FR2-022 - Existing hierarchical keywords shall be preserved by default.

FR2-023 - Unknown XMP namespaces shall be preserved.

FR2-024 - Existing develop/edit settings shall be preserved.

FR2-025 - Sidecar updates shall be merge-based, not full replacement.

FR2-026 - Sidecar writes shall be atomic: write temporary file, validate it, then replace the target (PW pattern, FR1-012d).

FR2-027 - Existing sidecars shall be backed up before modification when `--backup-sidecars` is enabled (default). Backups shall use a deterministic suffix (`.xmp.bak-<ISO-timestamp>`) and shall be referenced in the export report.

FR2-028 - After writing, the program shall re-read the sidecar and validate (a) that every expected keyword is present, and (b) via a field-level round-trip diff, that pre-existing flat keywords, hierarchical keywords, unknown namespaces, and develop settings all survived the merge. Presence of the new is half the contract; preservation of the old is the other half, and it shall be checked, not assumed.

FR2-028a - On validation failure: the backup shall be restored (when backups are enabled), the file marked `E_VALIDATION_FAILED`, the failure detailed in the export report, and the batch continued. A validation failure shall never leave the restored-from state ambiguous.

FR2-029 - Metadata writing shall go through a `MetadataWriteEngine` protocol. ExifTool is the first and only Phase 2 implementation. The protocol exists so Phase 4 packaging can swap engines without touching policy code; hand-rolled XMP merge logic is where metadata corruption comes from, and none shall be written in Phase 2.

FR2-029a - ExifTool shall be invoked in `-stay_open` batch mode. One process spawn per file is unacceptable at batch scale.

FR2-029b - ExifTool presence shall be verified at startup (`E_EXIFTOOL_MISSING` on failure, with installation guidance) and its version recorded in export provenance for every run; behavior differences across ExifTool versions are real.

FR2-029c - The engine wrapper shall expose dry-run rendering of intended changes so FR2-033 and the engine share one change-plan representation.

## 9. Output Requirements

FR2-030 - Phase 2 shall produce or update `.xmp` sidecars.

FR2-031 - Phase 2 shall preserve or create `.ai.json` sidecars unless `--no-write-ai-json` is given.

FR2-032 - Folder runs shall produce an export report (JSON, with the run's JSONL progress log as its substrate per FR1-012a) plus a human-readable summary.

FR2-033 - Dry-run mode shall show the complete intended XMP change plan — per file: tags to add, tags skipped and why, groups detected, conflicts — without writing sidecars.

FR2-034 - Export reports shall list files processed, sidecars created, sidecars modified, tags added with provenance, tags skipped with reasons (band threshold, specific-tag policy, normalization rejection), same-base-name groups and applied scope, conflicts, validation results, backups created, and errors with codes.

FR2-034a - Export reports shall end with post-export application instructions: Lightroom Classic does not re-read sidecars for already-imported photos automatically (the user must select the photos and invoke Metadata > Read Metadata from Files), and Capture One's sidecar synchronization depends on its Metadata preferences (Auto Sync sidecar XMP). The report shall state both, briefly, every time.

## 10. Acceptance Criteria

AC2-001 - The program can generate an XMP sidecar from one image using a fresh model run.

AC2-002 - The program can generate an XMP sidecar from an existing Phase 1 `.ai.json` sidecar.

AC2-003 - Accepted flat keywords appear in `XMP-dc:Subject`.

AC2-004 - Accepted hierarchical keywords appear in `XMP-lr:HierarchicalSubject` when enabled.

AC2-005 - Existing keywords are preserved after export, verified by the round-trip diff.

AC2-006 - Existing unknown XMP namespaces are preserved after export, verified by the round-trip diff.

AC2-007 - Proprietary RAW file hashes remain unchanged.

AC2-008 - A RAW+JPEG pair produces exactly one XMP sidecar containing the unioned, de-duplicated candidates with per-source provenance, and the group is reported.

AC2-009 - Dry-run mode produces a complete proposed-change plan without writing files.

AC2-010 - Exported sidecars can be read back by ExifTool, and the ExifTool version used is recorded in the report.

AC2-011 - A term containing `|` is rejected with a warning rather than exported or mangled.

AC2-012 - With defaults, no species-level or named-place tag is exported; with `--allow-specific-tags`, they are.

AC2-013 - A simulated validation failure restores the backup and the batch continues.

## 11. Future Groundwork

Phase 2 establishes the metadata-writing foundation for Phase 3 and Phase 4:

- the `MetadataWriteEngine` protocol and ExifTool batch wrapper;
- the XMP field mapper and keyword text rules;
- merge, backup, restore, and round-trip validation;
- same-base-name group resolution;
- conservative export policies keyed to confidence bands and specific-tag exclusion;
- dry-run change planning;
- export reporting with provenance.

Phase 3 shall extend this with batch-level tag normalization and folder-level subject consistency. Phase 4 shall wrap the same write/merge/validation layer in a GUI.
